# תוכנית עבודה סופית — מעבר Kung-Fu Chess לארכיטקטורת Client–Server

> מסמך מחייב המבוסס על מפרט המנהל, מבנה הפרויקט הקיים וכל 40 תשובות ההבהרה שאושרו.

## סטטוס מסירה ושער החלטה

- תוכנית זו היא מקור האמת למשימת המימוש החדשה.
- בדיקת הבסיס הושלמה: 518 בדיקות עברו וכיסוי הקוד הכולל הוא 90% בסביבת Python 3.11.1.
- אין לשנות את ה־API הקיים של `GameEngine` או לקבע את payload הסופי של `move_request`/`jump_request` לפני קבלת תשובת המרצה המופיעה בסוף המסמך.
- עד לקבלת התשובה יש להתקדם בכל המשימות שאינן תלויות בה: מעטפת פרוטוקול, קונפיגורציה, לוגים, SQLite, שני ה־tokens, `GameSession`, תור/worker ובדיקות תשתית.
- אם מתגלה סתירה, עמימות או החלטה שאינה כתובה במסמך, אין לבחור פתרון עצמאית. יש להסביר את האפשרויות וההשפעה ולבקש החלטה מבעלת הפרויקט.

## עמוד 1 מתוך 5 — יעד, היקף וארכיטקטורה

### 1. מטרת הפרויקט

מטרת הפרויקט היא להעביר את משחק Kung-Fu Chess הקיים מיישום מקומי הפועל בתהליך יחיד למערכת Client–Server מופרדת. המשחק נשאר משחק בזמן אמת: כלים של שני השחקנים יכולים לנוע בו־זמנית, לכלים יש מצבי תנועה, קפיצה ומנוחה, והמשחק מסתיים בעיקר בלכידת מלך. אין להפוך אותו לשחמט רגיל המבוסס על תורות.

השרת יהיה מקור האמת היחיד. הוא יחזיק את מצב המשחק, יקדם את הזמן, יאמת פעולות, יכריע לכידות ותוצאות, יקצה צבעים, ינהל משתמשים, שידוכים, חדרים, צופים, Sessions ו־Elo. הלקוח יציג את המשחק ב־OpenCV, יאסוף קלט וישלח בקשות בלבד. אין לאפשר ללקוח לעדכן בעצמו מצב משחק או דירוג.

הפיתוח הראשוני מתבצע מול שרת מקומי ב־Windows. כתובת השרת, הפורט וכל מגבלות המערכת ייקראו מקונפיגורציה, כדי שניתן יהיה להעביר בעתיד את השרת ל־Linux, לרשת מקומית או לאינטרנט. `ws` מותר בסביבה המקומית; לפני חשיפה לאינטרנט יהיה צורך ב־`wss/TLS` ובהקשחת אבטחה.

### 2. החלטות מוצר מחייבות

- פרוטוקול התקשורת הראשי יהיה JSON מובנה עם `type`, ‏`request_id`, ‏`game_id` ומזהה כלי יציב `piece_id`. המחרוזת `WQe2e5` יכולה להישמר לתאימות או לדמו בלבד.
- השרת מעבד פקודות לפי סדר הגעה סמכותי. פקודה חדשה לאותו כלי עסוק תידחה עם `piece_busy`; כלים אחרים רשאים לפעול במקביל אם הפעולה חוקית.
- השחור רואה לוח מסובב ב־180 מעלות בצד הלקוח בלבד. השרת שומר קואורדינטות קנוניות. צופים רואים כברירת מחדל את נקודת המבט של לבן.
- כל ממשקי `Login`, ‏`Registration`, ‏`Play`, ‏`Room`, שדה קוד חדר, שמות, דירוגים, שגיאות וספירת ניתוק יוצגו בתוך חלון OpenCV. לא ייפתח dialog חיצוני של Windows.
- הממשק יתמוך באנגלית ובעברית; עברית תוצג ב־RTL. השרת יחזיר קודי תוצאה ושגיאה, והלקוח יתרגם אותם מקובצי לוקליזציה.
- אין שמע ב־MVP. מערכת מצבי האנימציה הקיימת נשמרת, ללא דרישה חדשה לאנימציות התחלה וסיום.
- משחקי `Play` דרך matchmaking הם מדורגים; משחקי `Room` הם ידידותיים ואינם משפיעים על Elo.
- מגבלת חדר היא שני שחקנים ועד 10 צופים. היכולת לתמוך בהרבה משתמשים תושג באמצעות חדרים ומשחקים רבים במקביל.

### 3. עקרונות הנדסיים

1. **DRY:** חוק משחק, חישוב Elo או כלל הרשאה ימומשו במקום אחד בלבד.
2. **SRP:** לכל מודול אחריות אחת; לדוגמה `EloService` מחשב דירוג בלבד ו־`RoomManager` מנהל חדרים בלבד.
3. **ללא hard-coded constants:** כתובות, פורטים, זמני 20/60 שניות, טווח Elo, K-factor, מגבלות חדרים וטקסטים יהיו בתצורה.
4. **Encapsulation:** אין לחשוף collections פנימיים, חיבור SQLite או אובייקטי מנוע ישירות. משתמשים ב־interfaces, DTOs ו־repositories.
5. **Idempotency:** בקשות בעלות `request_id` זהה לא יבוצעו פעמיים. גם תוצאת משחק ועדכון Elo יוחלו פעם אחת בלבד.
6. **Server authoritative:** כל שעון, snapshot, תוצאה והרשאה נקבעים בשרת.
7. **Privacy by design:** סיסמאות, tokens, טלפון ומייל אינם נשלחים באירועי משחק ואינם נכתבים ליומנים.
8. **בידוד המנוע:** מנוע המשחק ומודולי ה־Domain אינם מכירים משתמשים, חדרים, SQLite, WebSocket, JSON או OpenCV. שכבת היישום בלבד מתרגמת בין אירועי המנוע לבין שירותי המערכת.
9. **Single Writer לכל משחק:** לכל `GameSession` יש `asyncio.Queue` ו־worker יחיד. כל שינוי במנוע עובר בתור, ולכן אין lock רחב סביב `GameEngine`. נעילה קצרה מותרת רק להקצאת מושבים או registry מחוץ למנוע.
10. **זמן סמכותי:** hybrid authoritative tick מקדם משחק פעיל גם ללא פקודות לקוח; השרת בלבד קובע מתי פעולה מתוזמנת הסתיימה.
11. **סנכרון ניתן לתיקון:** כל שינוי מקבל `sequence`; פער ברצף גורר בקשת snapshot מלא.
12. **התמדה idempotent:** constraints ו־transactions מונעים תוצאה או עדכון Elo כפולים.

### 4. מבנה היעד

```text
OpenCV Client
  UI/View -> Client Controller -> WebSocket Client -> JSON Protocol
                                                    |
                                                    v
Authoritative Server
  WebSocket Gateway -> Authentication/Session -> Message Router
                                         |-> Matchmaking Service
                                         |-> Room Manager
                                         |-> Game Session -> Existing Game Engine
                                         |-> Elo Service -> Repositories -> SQLite
                                         |-> Reconnect Manager
```

הרכיבים המרכזיים יהיו `ConfigProvider`, ‏`SchemaValidator`, ‏`WebSocketGateway`, ‏`MessageRouter`, ‏`AuthService`, ‏`SessionManager`, ‏`MatchmakingService`, ‏`RoomManager`, ‏`GameSession`, ‏`ReconnectManager`, ‏`EloService`, ‏`UserRepository` ו־`MatchRepository`. מנוע המשחק לא יכיר OpenCV, WebSocket, JSON או SQLite.

האירועים יחולקו לשלוש שכבות: אירועי Domain של מנוע משחק יחיד, אירועי Application הכוללים הקשר כגון `game_id`, והודעות JSON הנשלחות ברשת. כחלק מכך יתווסף למנוע `GameOverEvent`, שיפורסם פעם אחת בדיוק כאשר המשחק עובר ממצב פעיל למצב שהסתיים.

## עמוד 2 מתוך 5 — תשתית, Message Bus והפרדת לקוח–שרת

### שלב 0 — קיבוע חוזים ורשת ביטחון

**מטרה:** להכין בסיס שניתן לפתח עליו במקביל באמצעות Agents בלי ליצור מימושים סותרים.

**משימות משותפות**

1. למפות את זרימת הקלט הנוכחית: OpenCV → `Game` → `GameEngine` → `GameSnapshot` → ציור.
2. לשמר את מנוע Kung-Fu Chess הקיים ואת מצבי `idle`, ‏`move`, ‏`jump`, ‏`short_rest`, ‏`long_rest` כמקור אמת לחוקי המשחק.
3. ליצור קובץ תצורה חיצוני עם schema עבור כתובת/פורט, tick rate, timeout שידוך, grace period, Elo, מגבלות חדרים, לוגים ונתיבי SQLite וגיבוי.
4. להגדיר version לפרוטוקול, envelope אחיד וקודי שגיאה יציבים.
5. לכתוב בדיקות characterization למנוע הקיים: תנועה בו־זמנית, כלי עסוק, לכידה, לכידת מלך, שני מלכים באותו tick ויצירת snapshot.
6. להוסיף בדיקה ש־`GameOverEvent` מתפרסם פעם אחת בלבד כאשר תנאי הסיום מתקיים, ואינו מתפרסם שוב ב־ticks מאוחרים.
7. לחלק עבודת Agents רק לאחר קיבוע interfaces. Agent אחד לא ישנה חוזה משותף ללא עדכון מתועד ובדיקת אינטגרציה.

**תוצר:** חוזים משותפים, קונפיגורציה, בדיקות מנוע בסיסיות ורשימת קודי אירועים ושגיאות.

### שלב 1 — הפשטת ערוץ התקשורת

**מטרה:** לנתק את פקודות ה־UI מלוגיקת המשחק לפני הכנסת רשת.

**Backend / Domain**

- להרחיב את ה־Event Bus הקיים לחוזים מטיפוס מוגדר, בלי להפוך אותו לתחליף ל־WebSocket.
- להפריד בין Commands המבקשים שינוי, Events המתעדים מה שאירע ו־Snapshots המתארים מצב קריאה.
- להוסיף `GameOverEvent` כאירוע Domain המכיל את המנצח ואת זמן הסיום הסמכותי. `GameEngine` יפרסם אותו בנקודה היחידה שבה המשחק משתנה ל־game over.
- `GameService` יתרגם את `GameOverEvent` לאירוע Application בשם `GameEndedEvent`, הכולל `game_id`, סיבת סיום ותוצאה. רכיב הרשת יתרגם אותו להודעת JSON מסוג `game_over` לכל השחקנים והצופים.
- להגדיר handlers יחידים לפעולות משחק ולוודא שכל שינוי מצב עובר דרך ה־GameEngine.
- למנוע גישה ישירה של UI לשדות פנימיים במנוע.

**Frontend**

- להכניס `ClientController` בין אירועי עכבר/מקלדת לבין המשחק.
- בשלב זה ניתן לחבר אותו למתאם מקומי המדמה שרת, כדי לוודא שה־UI אינו תלוי במימוש המנוע.
- להציג אישור או דחייה לפעולה לפי קוד תוצאה, ולא לפי שינוי מקומי אופטימי שאינו מאושר.

**Patterns:** Command, Observer/Event Bus, Adapter, immutable DTO, Ports and Adapters.

**סיכונים ובדיקות:** מניעת פרסום אירוע כפול, שמירת סדר אירועים, אי־שינוי snapshot לאחר פרסומו, והבטחה שהוצאת ה־UI מהתהליך לא משנה את חוקי המשחק.

### שלב 2 — WebSocket ותהליכים נפרדים

**מטרה:** להריץ שרת ולקוחות בתהליכים נפרדים ולסנכרן לפחות שני לקוחות בזמן אמת.

**Backend**

1. להקים שרת WebSocket אסינכרוני מקומי ללא UI.
2. לממש `ConnectionRegistry` ו־`MessageRouter` בלי לקשור socket ישירות ל־GameEngine.
3. לאמת כל JSON מול schema: גרסת פרוטוקול, סוג הודעה, שדות חובה, ערכים וגדלים.
4. ליצור `GameSession` המחזיק מנוע משחק יחיד, שעון סמכותי, שני מושבים ורשימת צופים.
5. להפעיל עבור כל משחק `asyncio.Queue` ו־worker יחיד. move, jump, tick, disconnect, reconnect ו־timeout ייכנסו לאותו תור; רק ה־worker ישנה את `GameEngine`.
6. לעבד `move_request` לפי סדר הגעה; לדחות משתמש שאינו בעל הכלי, משחק לא מתאים, יעד בלתי חוקי או `piece_busy`.
7. להפעיל tick סמכותי תקופתי שניתן להגדרה עבור משחקים שיש בהם תנועה, קפיצה או cooldown פעילים, כדי שהזמן יתקדם גם כאשר לא מגיעה פקודת לקוח.
8. לשדר `command_result`, עדכונים ו־snapshots עם sequence number. לקוח שמזהה פער יבקש resync.
9. להגריל לבן/שחור ב־Play. בחדר, היוצר לבן והמצטרף הראשון שחור.

`sequence` הוא מספר גרסת המצב שעולה לאחר כל שינוי סמכותי. אם הלקוח קיבל, לדוגמה, גרסה 41 ולאחריה 43, הוא יודע שעדכון חסר ואינו מנסה להשלים אותו בעצמו. הוא מבקש snapshot — צילום מלא ועדכני של הלוח, הכלים, התנועות, ה־cooldowns ומצב המשחק — ומחליף באמצעותו את מצב התצוגה המקומי.

הקוד הקיים כבר מעניק לכל כלי `piece_id` מספרי יציב בתוך המשחק, שומר אותו בתנועה ובקידום ומפרסם אותו ב־`GameSnapshot`. הודעת רשת תכיל `game_id`, ‏`piece_id`, ‏`expected_from` ו־`target`. `GameService` יאתר את הכלי מתוך snapshot, יאמת צבע, מצב ומיקום, ויתרגם לקואורדינטות שה־`GameEngine` הקיים דורש. אין צורך לשנות את מודל `Piece`.

**Frontend**

- להחליף את המתאם המקומי ב־WebSocket client.
- לשמור state תצוגה בלבד ולהחליפו לפי snapshots שאושרו בשרת.
- לסובב את תצוגת השחור בלי לשנות את הקואורדינטות הנשלחות.
- להציג מצבי connecting, connected, reconnecting ו־error.

**Database:** עדיין אין תלות הכרחית ב־SQLite; ניתן להשתמש בזהויות בדיקה זמניות בלבד בשער שלב זה.

**תנאי מעבר:** שני לקוחות נפרדים רואים אותו משחק; פקודה מזויפת נדחית; בקשה כפולה אינה מבוצעת פעמיים; resync מתקן לקוח שאיבד עדכון.

## עמוד 3 מתוך 5 — משתמשים, SQLite, Elo ו־Matchmaking

### שלב 3 — הרשמה, Login ו־Persistence

**מטרה:** לזהות משתמשים בצורה מאובטחת יחסית לסביבת MVP מקומית ולשמור דירוג ותוצאות באופן עקבי.

**Backend**

- לממש רישום עצמי ולאחריו Login מפורש.
- `username` יהיה ייחודי, באורך 3–20, ויכיל אותיות אנגליות, מספרים וקו תחתון. הוא תלוי רישיות: `Dana` ו־`dana` הם חשבונות שונים. שם פרטי, אם נשמר, אינו מזהה חשבון.
- סיסמה תהיה באורך 6 תווים לפחות ותישמר באמצעות password hash עם salt; לעולם לא כטקסט רגיל.
- לאחר Login השרת ינפיק Authentication token. הערך הגלוי יימסר ללקוח, ורק hash שלו יישמר ב־SQLite עם תוקף וביטול הניתנים להגדרה.
- עם הצטרפות למשחק השרת ינפיק Game token נפרד, הקשור ל־`game_id`, למשתמש ולמושב. גם הוא יישמר ב־SQLite כ־hash בלבד.
- Authentication token מוכיח זהות כללית; Game token מוכיח בעלות על מושב במשחק מסוים. reconnect מחייב את שניהם.
- Game token מתבטל בסיום או בביטול המשחק, ב־resign, או לאחר 20 שניות ניתוק ללא reconnect.
- restart ללא active-game recovery מסמן משחקים פעילים interrupted ומבטל את Game tokens שלהם; Authentication token יכול להישאר תקף בהתאם לתוקפו.
- איפוס באמצעות התאמת מספר טלפון הוא פתרון זמני לפיתוח המקומי בלבד. בשלב מתקדם יישלח קוד זמני למייל עם תוקף והגבלת ניסיונות.

**Frontend**

- ליצור בתוך OpenCV מסכי Registration, ‏Login, הודעות validation ומעבר למסך הראשי.
- לא להציג או לשמור סיסמה גלויה לאחר השליחה.
- לשמור token רק במיקום לקוח מוגן ככל האפשר ולהסירו ב־logout או כאשר השרת דוחה אותו.

**Database**

- להשתמש ב־wrapper/repository וב־migrations; אין להריץ SQL מתוך UI או handlers של WebSocket.
- טבלת משתמש תכלול מזהה פנימי, username, password hash, מייל, טלפון, Elo התחלתי 1200, סטטוס ותאריכי יצירה/עדכון.
- טבלת `auth_sessions` תשמור hash של Authentication token, בעלים, תוקף ומצב ביטול.
- טבלת `game_session_tokens` תשמור hash של Game token, ‏`game_id`, בעלים, צבע/תפקיד, סטטוס, `grace_expires_at` וביטול.
- טבלת `rooms` תשמור metadata של חדר: קוד, יוצר, סטטוס, זמני יצירה/התחלה/סגירה וסיבת סיום.
- טבלת `room_members` תשמור חברות היסטורית: משתמש, תפקיד, צבע, זמני הצטרפות ועזיבה. אין לשמור בה socket חי.
- טבלת תוצאות מדורגות תכלול מזהה משחק ייחודי, שחקנים, תוצאה, Elo לפני/אחרי וסיבת סיום.
- מייל וטלפון פרטיים: אינם מוצגים, אינם נשלחים במשחק ואינם נרשמים בלוגים.
- במחיקת חשבון, פרטי הזיהוי והכניסה יימחקו או יעברו אנונימיזציה; תוצאות משחקים יישארו אנונימיות כדי לשמור על עקביות היסטורית.

**גיבוי והרשאות**

- ליצור גיבוי SQLite לפני עדכון גרסה משמעותי ובאופן אוטומטי אחת ליום.
- תוצאות יישמרו ללא הגבלת זמן, אלא אם תוגדר מדיניות חדשה.
- רק מנהל מערכת רשאי לשחזר גיבוי או למחוק נתוני מערכת; הפעולה תירשם ב־audit log.

### שלב 4 — Elo ו־Play Matchmaking

**מטרה:** לחבר שני משתמשים מתאימים למשחק מדורג ולשמור עדכון Elo יחיד ונכון.

**Elo**

- דירוג התחלתי `1200`, ‏scale של `400`, ‏K-factor קבוע `32`, רצפת דירוג `100`.
- ציון תוצאה: ניצחון `1`, הפסד `0`, תיקו טכני `0.5`.
- העיגול יהיה Half-Up: פחות מ־`.5` למטה, ו־`.5` ומעלה למעלה.
- אין הצעת תיקו. תיקו ייקבע רק על ידי השרת, למשל לכידת שני מלכים באותו tick או סיום טכני ללא מנצח.
- תוצאת המשחק ועדכון שני הדירוגים יישמרו ב־transaction יחיד ובאמצעות מזהה משחק ייחודי.
- טבלת שינויי הדירוג תכיל constraint ייחודי המונע החלת תוצאה נוספת עבור אותו שילוב של משחק ומשתמש. בקשה חוזרת תחזיר את התוצאה שכבר נשמרה ולא תחשב Elo מחדש.

**Matchmaking Backend**

1. `Play` יוצר ticket יחיד למשתמש מאומת שאינו במשחק או בתור.
2. טווח ההתאמה הוא ±100 Elo כולל, ונשאר קבוע במשך 60 שניות.
3. כאשר יש כמה מועמדים מתאימים, נבחר הכרטיס הוותיק ביותר.
4. לבן ושחור יוגרלו בשרת.
5. `Cancel` או ניתוק בזמן ההמתנה מסירים את הכרטיס ללא הפסד וללא Elo.
6. timeout לאחר 60 שניות מחזיר קוד שגיאה מקומי ומסיר את הכרטיס.

**כללי דירוג לסיום מוקדם**

- רק Play מדורג. Room אינו מדורג.
- ניתוק יחיד ייחשב הפסד מדורג רק אם המשחק התחיל, השחקן המנותק כבר הזיז לפחות כלי אחד, הוא לא חזר בתוך 20 שניות והשרת הכריז `forfeit`.
- `resign` ידני, אם יתווסף, ישפיע על Elo רק לאחר שהפורש ביצע לפחות פעולה חוקית אחת.
- אם שני השחקנים מתנתקים, המשחק ימשיך רק אם שניהם חוזרים בזמן. אם רק אחד או אף אחד לא חזר, המשחק מתבטל ללא Elo.
- נפילת שרת או restart מבטלים משחק ללא Elo ב־MVP.

**תנאי מעבר:** בדיקות נוסחה וגבולות, מניעת ticket כפול, התאמת 1100–1300 לשחקן 1200, timeout מדויק, ביטול בטוח ועדכון Elo פעם אחת בלבד.

## עמוד 4 מתוך 5 — Rooms, Spectators, Reconnect וממשק מלא

### שלב 5 — חדרים פרטיים וצופים

**מטרה:** להפריד בין `Play` מדורג לבין `Room` ידידותי, עם הרשאות ברורות לשחקנים ולצופים.

**Backend**

- `Create Room` יוצר קוד אקראי בן שישה תווים מאותיות ומספרים, ללא `O`, ‏`0`, ‏`I`, ‏`1`. הקוד אינו תלוי רישיות והשרת מנרמל אותו ל־uppercase.
- יוצר החדר תופס את המושב הלבן. המצטרף הראשון תופס שחור. משתמשים נוספים הם צופים עד מגבלת 10.
- אין ב־MVP חדר ללא מארח משחק, סיסמת חדר או קישור הזמנה. קישור הזמנה במייל שייך לשלב מתקדם.
- צופה יכול להצטרף גם לאחר תחילת המשחק ויקבל תחילה snapshot מלא ולאחריו עדכונים מיידיים.
- צופה אינו רשאי לשלוח פקודות משחק, לראות מידע פרטי או להתקדם למושב במהלך משחק פעיל.
- לאחר סיום המשחק אין הצטרפות חדשה. הצופים הקיימים רואים את התוצאה, והחדר נסגר לאחר זמן המוגדר בקונפיגורציה.
- אם היוצר עוזב לפני תחילת המשחק, החדר נסגר עם `room_closed`. אם שחקן אחר עוזב לפני ההתחלה, ניתן לשחרר את מושבו למצטרף הבא.

**Frontend**

- לחיצה על `Room` תפתח בתוך OpenCV תיבה הכוללת קוד בן שישה תווים, `Join`, ‏`Cancel` ושגיאות.
- מסך יצירת חדר יציג את הקוד ואת מצב ההמתנה לשחקן שחור.
- תצוגת צופה תהיה read-only, מנקודת מבט לבן, עם שמות, דירוגים ותוצאת סיום.
- כל קודי `room_not_found`, ‏`room_full`, ‏`room_closed`, ‏`already_in_game` ו־`invalid_room_code` ימופו לטקסט עברי/אנגלי.

**Database:** המערכת תשתמש במודל משולב. metadata והיסטוריה של חדרים וחבריהם יישמרו בטבלאות `rooms` ו־`room_members`. חיבורי WebSocket, ‏`GameSession`, ‏`GameEngine`, תורים, timers ומצב משחק חי יישארו בזיכרון. שמירת רשומת חדר אינה מאפשרת recovery של משחק. לאחר restart, חדר שהיה `WAITING` או `ACTIVE` יסומן `INTERRUPTED`/`CLOSED`, ה־Game tokens יבוטלו והמשחק לא ישפיע על Elo.

### שלב 6 — ניתוק וחיבור מחדש

**מטרה:** לאפשר חזרה בטוחה למשחק בלי לתת יתרון לשחקן המחובר ובלי לאפשר גנבת מושב.

**ניתוק שחקן יחיד**

1. השרת מעביר את המשחק ל־`grace_pause` למשך 20 שניות.
2. אין לקבל פעולות חדשות מאף שחקן בזמן זה. פעולות שכבר אושרו יכולות להסתיים סמכותית.
3. המושב נשמר לאותו משתמש; צופה אינו יכול לתפוס אותו.
4. הלקוח המחובר מציג ספירה לאחור שמקורה בזמן השרת.
5. reconnect שולח Authentication token, ‏Game token, ‏`game_id` ו־`request_id`.
6. השרת גוזר את זהות המשתמש מה־Authentication token, מאמת ש־Game token שייך לאותו משתמש ולמשחק, בודק בעלות על המושב ותוקף חלון החסד, ואז שולח snapshot עדכני.
7. אם המשתמש לא חזר, השרת מחיל את כללי ה־forfeit והדירוג של שלב 4.

סגירה יזומה של האפליקציה נחשבת ניתוק ומקבלת 20 שניות; רק כפתור `Resign` מפורש, אם ימומש, מסיים מיד.

**ניתוק שני שחקנים**

- שני המושבים נשמרים והמשחק מושהה ל־20 שניות.
- המשחק ממשיך רק אם שני המשתמשים חזרו ואומתו בזמן.
- אם רק אחד חזר או שאיש לא חזר, המשחק מתבטל ללא מנצח, מפסיד או Elo.
- שחקן יחיד שחזר יקבל הודעה שהמשחק בוטל עקב ניתוק כפול.

**נפילת שרת**

- משחק פעיל אינו נשמר ב־SQLite ב־MVP.
- restart מבטל את המשחק ללא Elo ומבטל את ה־Game tokens של משחקים פעילים. Authentication token תקף נשאר ניתן לשימוש; רק token שפג או בוטל מחייב Login מחדש.
- האירוע יירשם כ־`server_interrupted_game`.

### שילוב UI סופי

יש לממש state machine למסכי OpenCV: `LOGIN`, ‏`REGISTER`, ‏`HOME`, ‏`MATCHMAKING`, ‏`ROOM_LOBBY`, ‏`PLAYING`, ‏`SPECTATING`, ‏`GRACE_PAUSED`, ‏`RESULT`. מעבר מסך יתבצע לפי אירועי שרת, בלי תנאי UI מפוזרים. טקסטים יהיו בקובצי localization, כולל RTL, הודעות timeout ושגיאות authentication/room/reconnect.

**תנאי מעבר:** הצטרפות לפי תפקיד, חסימת פקודת צופה, snapshot לצופה מאוחר, חזרת אותו שחקן לאותו מושב, ביטול ללא Elo בניתוק כפול וסגירת חדר תקינה.

## עמוד 5 מתוך 5 — אמינות, בדיקות, סיכונים ומסירה

### שלב 7 — לוגים, קיבולת והקשחה

**לוגים**

- לקוח ושרת ישמרו JSON Lines עם `timestamp`, ‏`level`, ‏`event`, מזהים רלוונטיים ו־`request_id`.
- השמירה תהיה 14 ימים או 100MB לכל צד, המוקדם מביניהם, באמצעות rotation.
- payload מלא מותר רק ב־DEBUG מקומי וגם אז לאחר masking.
- אין לרשום סיסמה, hash, token, מייל או טלפון.
- אירועים קריטיים: התחברות, כשל authentication, תור, שידוך, יצירת חדר, הצטרפות, פקודה מאושרת/נדחית, reconnect, forfeit, ביטול, עדכון Elo, גיבוי ותקלה.

**יעדי קיבולת ראשוניים לתהליך יחיד**

- עד 50 משחקים פעילים.
- עד 200 משתמשים בתור.
- עד 100 חדרים פתוחים.
- עד 10 צופים בחדר.
- עד 500 חיבורי WebSocket בסך הכול.

המספרים הם יעדי תכנון ובדיקת עומס הניתנים להגדרה, ולא הבטחת production לפני מדידה. מגבלת 500 החיבורים היא התקרה הכוללת; אין דרישה שכל יתר ערכי המקסימום יתממשו יחד. אם נדרשים אלפי משתמשים, יש לעבור בהמשך למספר תהליכים/שרתים, state משותף ומסד נתונים שאינו SQLite.

### אסטרטגיית בדיקות

| שכבה | בדיקות חובה |
|---|---|
| Domain | תנועה בו־זמנית, כלי עסוק, לכידות, מלך, תיקו טכני, snapshot ו־`GameOverEvent` יחיד |
| Protocol | schema, version, הודעה לא מוכרת, שדה חסר, payload גדול, `request_id` כפול |
| Authentication | הרשמה, שם תלוי רישיות, hash, Login כפול, token שגוי/פג |
| SQLite | migrations, constraints, transaction Elo, אנונימיזציה, גיבוי ושחזור מורשה |
| Matchmaking | ±100 כולל, FIFO, timeout, cancel, disconnect ותחרות בין בקשות |
| Rooms | קוד מנורמל, מושבים, 10 צופים, הרשאות, סגירת מארח ו־late join |
| Reconnect | שחקן יחיד, שני שחקנים, חזרה בגבול הזמן, token מזויף ו־restart |
| UI | סיבוב שחור, RTL, מסכים, שגיאות, countdown ותצוגת צופה |
| Load | 500 חיבורים, 50 משחקים, תור גדול, broadcast לצופים וצריכת זיכרון |

כל שלב יסתיים ב־unit tests, integration tests ותרחיש demo. אין למזג שינוי Agent שלא עבר את הבדיקות הרלוונטיות ולא נבדק מול החוזים המשותפים.

### סיכונים מרכזיים ומענה

1. **מרוצי פקודות בזמן אמת:** להשתמש בלולאת משחק סמכותית ובסדר עיבוד מוגדר לכל GameSession.
2. **אירוע סיום כפול:** לפרסם `GameOverEvent` רק במעבר הראשון ל־game over ולבדוק idempotency בכל consumer.
3. **חוסר התאמה בין לקוחות:** sequence numbers, snapshot מלא ו־resync.
4. **Elo כפול או חלקי:** מזהה תוצאה ייחודי ו־transaction אחד.
5. **התחזות ב־reconnect:** token אקראי, תפוגה, אימות בעלות על מושב ואי־רישום token.
6. **חסימת event loop:** פעולות SQLite וגיבוי לא ירוצו כעבודה ארוכה בתוך לולאת הרשת.
7. **SQLite תחת עומס:** כתיבות קצרות, indexes, transactions ומדידה; מעבר עתידי למסד שרת אם נדרש.
8. **איפוס טלפוני חלש:** מוגבל לפיתוח מקומי בלבד; אין לחשוף אותו לאינטרנט. השלב הבא הוא קוד זמני במייל.
9. **ריבוי Agents:** interfaces קפואים לכל אבן דרך, בעלות קבצים ברורה, ענפים נפרדים ובדיקת אינטגרציה מרכזית.
10. **מחיקת משתמש מול היסטוריה:** אנונימיזציה של תוצאות במקום מחיקת הקשרים ההיסטוריים.

### סדר מסירה ותלויות

```text
חוזים ובדיקות מנוע
  -> Message Bus והפרדת UI
  -> WebSocket ותהליכים נפרדים
  -> Authentication + SQLite
  -> Elo + Matchmaking
  -> Rooms + Spectators
  -> Reconnect + Reliability
  -> Load, Security ו־Release
```

ניתן לבצע במקביל רק עבודה שאינה תלויה בהחלטה שעדיין משתנה. לאחר קיבוע חוזה הפרוטוקול ניתן לפתח במקביל את שרת הרשת, לקוח OpenCV ו־SQLite repositories. לאחר קיבוע שירותי המשתמש והמשחק ניתן לפתח במקביל Matchmaking, Rooms ובדיקות עומס. בעלת הפרויקט היא המאשרת הסופית של כל אבן דרך.

### קריטריוני קבלה סופיים

הפרויקט ייחשב מושלם כאשר שני משתמשים יכולים להירשם ולהתחבר, לבחור `Play`, להשתדך לפי הכללים, לקבל צבעים, לשחק בזמן אמת משני תהליכים נפרדים ולקבל עדכון Elo יחיד ונכון. בנוסף, משתמש יכול ליצור `Room`, לצרף יריב ועד 10 צופים, והצופים מקבלים מצב מיידי בלי יכולת לשחק. ניתוק מטופל בספירת 20 שניות לפי כללי יחיד/כפול, כל המידע מוצג בתוך OpenCV בעברית או באנגלית, restart אינו גורם לעדכון Elo שגוי, המידע הפרטי אינו נחשף, הלוגים והגיבויים פועלים, וכל בדיקות הקבלה והעומס שנקבעו עוברות.

### מחוץ ל־MVP

Replay והיסטוריית מהלכים מלאה, persistence של מצב משחק חי ו־recovery של חדר/משחק לאחר restart, קישור הזמנה במייל, spectator delay תחרותי, K-factor דינמי, לקוח Linux/macOS ופריסה ציבורית עם TLS יישמרו לשלבים מתקדמים ולא יעכבו את ה־MVP המקומי.

### שאלת המרצה — שער לפני מימוש פקודות כלי

> במימוש הקיים, לכל כלי כבר יש `piece_id` מספרי ויציב שמופיע ב־`GameSnapshot`, אבל הממשק הציבורי של `GameEngine` מקבל קואורדינטות באמצעות `request_move(from_row, from_col, to_row, to_col)`. האם מקובל להשאיר את מנוע המשחק והיישום המקומי ללא שינוי, ולהוסיף בשכבת השרת Adapter שמקבל הודעת רשת הכוללת `game_id`, ‏`piece_id`, ‏`expected_from` ו־`target`, מאתר את הכלי ב־snapshot, מאמת בעלות, מצב ומיקום ומתרגם את הבקשה לקריאה הקיימת לפי קואורדינטות? או שנדרש לשנות את ה־API של `GameEngine` כך שיקבל `piece_id` ישירות? ההעדפה הארכיטקטונית היא Adapter, כדי לשמור על backward compatibility, הפרדת אחריות ואי־תלות של מנוע המשחק ברשת. האם פתרון זה עומד בדרישות המטלה?

עד לקבלת תשובה אין לשנות את `GameEngine`, לממש `NetworkGameAdapter` או לקבע את חוזה `move_request`/`jump_request`. לאחר קבלת התשובה יש להציג אותה לבעלת הפרויקט, לעדכן את המסמך ורק אז להמשיך בחלקים אלה.
