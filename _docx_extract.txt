Kung Fu Chess — Design Guide: Implementation and Tests
1. Purpose of This Document
This document guides the teacher in leading students through the design and implementation of a Kung Fu Chess game using disciplined, test-driven, AI-assisted software engineering.
The goal is not only to finish a game. The real goal is to train students to:
Break a vague game idea into small, testable requirements.
Separate model, controller, real-time game logic, and rendering.
Write unit tests before implementation.
Write text-based integration tests that validate visible behavior.
Use an LLM as a junior development assistant, not as an uncontrolled authority.
Refactor safely under test coverage.
Explain design decisions and prove correctness through tests.
The teacher uses this guide to steer the class. Students do not receive the full architecture as a finished solution on day one. The teacher reveals the next design boundary and the next test requirement at each iteration.
2. Course-Level Engineering Rules
Every development step follows the same loop:
Requirement → test plan → failing tests → implementation → refactor → AI review → peer review → commit
This loop is more important than the specific game feature. The teacher stops students who jump directly into implementation without first making the behavior observable through tests.
Required Engineering Habits
Students must:
Keep the game logic independent of the drawing library.
Write unit tests for pure logic before connecting UI.
Use fake clocks or explicit elapsed-time calls instead of real sleeping in tests.
Keep rendering thin and replaceable.
Avoid hidden global state.
Use small classes with clear responsibility.
Commit only when tests pass.
Include a short explanation of each nontrivial design decision.
AI-Assisted Development Habit
When using an LLM, students ask it for small, reviewable steps:
“Write tests for this requirement.”
“Propose the smallest class boundary for this behavior.”
“Review this diff for responsibility leakage.”
“Find missing edge cases.”
“Explain why this test fails.”
Prompt discipline examples:
Bad: “Build the whole movement system.” Better: “Given the requirement that a rook moves horizontally and vertically until blocked, propose five unit tests before writing implementation.”
Bad: “Fix my code.” Better: “Here is a failing test and the relevant class. Explain the likely boundary issue before suggesting a minimal change.”
Bad: “Design the architecture.” Better: “Compare putting pixel-to-cell mapping in Controller versus BoardMapper. Decide which keeps the model cleaner and why.”
Students do not ask the LLM to “build the whole game.” The teacher treats the LLM like a fast but unreliable junior developer: useful, but always reviewed by tests and humans.
Students first write their own short plan or diagnosis before asking the LLM to respond. The LLM may critique the plan, find missing cases, or compare alternatives, but it does not replace the student’s design responsibility.
Before accepting help, students ask the LLM to restate what it understood. If the restatement is wrong or vague, students correct the request before asking for code.
Students work one step at a time: first clarify the requirement, then choose the responsibility boundary, then write tests, then implement the smallest change, then review the diff.
When the goal is learning, students ask for a clue, a failing test, or the next diagnostic question instead of asking for the full answer.
AI Debugging Protocol
When debugging with an LLM, students give exact reproduction steps, the expected behavior, the actual behavior, and the smallest relevant code or test. They also state which layer they suspect: Controller, GameEngine, RuleEngine, RealTimeArbiter, Board, Renderer, or TextTestRunner.
Before asking for a fix, students ask the LLM to identify which layer is most likely responsible and what evidence would confirm or reject that diagnosis.
If the LLM proposes several changes at once, students ask it to choose the smallest first change and the test that should fail before the change and pass after it.
Students ask the LLM to locate the likely responsibility boundary before asking for code. They must ask the LLM to explain why any suggested change is needed.
Students treat large unexplained rewrites as suspicious. If the LLM replaces a large amount of code with a small patch, students ask whether behavior was intentionally removed and which tests prove it remains covered.
After receiving an answer, students may ask the LLM to review its own answer for possible mistakes. The teacher still requires human review and passing tests.
3. Scope of the Game
Kung Fu Chess is a real-time chess-like game. Pieces move according to chess movement patterns, but moves are not instantaneous. A player selects a piece and clicks a destination. The piece travels toward that destination at a fixed speed.
For this course, the rules are fixed and intentionally simplified.
Common Route Rules
The common route is the required path for all students.
The board is any rectangular size inferred from a textual description.
Pieces are described using normal chess letters: K, Q, R, B, N, P.
Piece color is encoded by prefix: wK, bR, etc.
. means an empty square.
The game does not implement check, checkmate, castling, en passant, or promotion.
A king can be captured.
Capturing the opponent king ends the game.
A piece moves only according to its movement rule.
Sliding pieces do not pass through blocking pieces.
Movement has a fixed speed.
Tests use a fixed cell size in pixels.
UI rendering uses only the supplied drawing/image library.
The logical board changes only when a moving piece arrives.
The common route allows only one active motion at a time. If a legal-looking move is requested while a motion is already active, GameEngine rejects the request with reason "motion_in_progress". No new motion starts and Board remains unchanged.
The text integration DSL contains only Board, click, wait, and print board.
print board is the only integration-test assertion mechanism.
Extra Route Rules
The extra route is for students who finish the common route cleanly. The teacher chooses one extra feature per strong team. The allowed extra features are:
Simultaneous movement of multiple pieces.
Cooldown after movement.
Collision between moving pieces.
Cancellation when the destination becomes occupied before arrival.
Replay file support.
Basic bot strategy.
Visual polish without changing the model.
The extra route must not damage the common route architecture. An extra feature is architecturally acceptable only when it extends the appropriate layer without reversing dependency direction or merging unrelated responsibilities. Cooldown belongs in movement or game-state coordination, not in the renderer. Replay belongs around command or event recording, not inside Board. Visual polish belongs in the renderer, not in the rules layer. If students cannot add the feature without rewriting the common route, the design is rejected and refactored first.
4. Core Design Decision: Split the System Into Testable Layers
The implementation is split into these layers:
Model — Board, positions, pieces, piece colors, piece types, and piece states.
Movement rules — Legal destinations for each kind of piece.
Rule engine — Validation of requested moves against the current board state.
Real-time arbiter — Movement over time, active motions, arrival resolution, capture events, and king-capture reporting.
Controller — Translation of clicks into game commands.
View / renderer — Drawing the current state using the supplied drawing library.
Text I/O — Parsing textual board setup and printing logical board state.
Text test runner — Execution of textual integration tests without a real UI.
Every layer must be testable without the layers above it. Layer ownership is fixed:

6. Model Design
Position
Position represents a board cell, not pixels.
Fields:
row: intcol: int
Required behavior:
Equality.
Readable representation.
No board-bounds checking inside Position.
Board bounds belong to Board.
Position is a value object. It does not know board size, rendering, movement rules, or input coordinates.
Unit tests:
Two positions with the same row and column are equal.
Two positions with different row or column are not equal.
Position objects produce readable assertion failures.
Piece
Piece represents a chess piece.
Fields:
id: unique stable idcolor: white / blackkind: king / queen / rook / bishop / knight / pawncell: Positionstate: idle / moving / captured
Piece.state is only a lifecycle flag: idle, moving, or captured. It does not store path, destination, elapsed time, speed, interpolation, or arrival logic. Those details belong to Motion and RealTimeArbiter.
Piece IDs are assigned consistently at creation time, either by BoardParser or by a dedicated PieceFactory. If stable identity is used for motion tracking or snapshots, duplicate piece IDs are invalid.
A piece never knows about the renderer, mouse clicks, pixels, or text test syntax.
Board
Board owns the logical arrangement of pieces.
Responsibilities:
Store width and height.
Add a piece.
Remove a piece.
Query a piece at a cell.
Check whether a cell is inside bounds.
Move a piece after a move has been validated.
Reject duplicate occupancy.
Board does not contain chess movement rules. The board knows what exists; it does not decide which chess moves are legal.
Board.move_piece assumes validation has already happened. Board does not call RuleEngine, and RuleEngine does not mutate Board.
Unit tests:
Board dimensions are inferred correctly.
Empty cells return no piece.
Occupied cells return the correct piece.
Adding two pieces to the same cell fails.
Moving a piece updates source and destination.
Removing a captured piece clears its cell.
7. Movement Rule Design
Each piece type has a small rule class with this interface:
legal_destinations(board, piece) -> set[Position]
The method returns destinations based on piece movement and blocking. Enemy-occupied destinations may be legal destinations, but the rule class does not capture, remove, move, or mutate pieces.
Common Movement Rules
Rook: Horizontal and vertical sliding until blocked.
Bishop: Diagonal sliding until blocked.
Queen: Rook movement plus bishop movement.
Knight: L-shaped jumps, ignoring blockers.
King: One square in any direction.
Pawn: Simplified forward movement.
Pawn behavior is fixed:
White pawn moves one row upward.
Black pawn moves one row downward.
Pawn captures one diagonal step forward.
Pawn has no initial two-step move.
Pawn has no en passant.
Pawn has no promotion.
PieceRules are stateless. They do not store selected pieces, active motions, elapsed time, or game-over state. They only calculate legal destinations from the given board and piece.
The implementation order is fixed:
Rook.
Bishop.
Queen.
Knight.
King.
Pawn.
Unit tests are written per piece type:
Rook moves across empty row and column.
Rook stops before a friendly blocker.
Rook captures an enemy blocker but does not pass it.
Bishop moves diagonally and not straight.
Queen combines rook and bishop movement.
Knight jumps over blockers.
King moves one cell only.
Pawn moves and captures according to the simplified pawn rules.
8. RuleEngine Design
RuleEngine answers this question:
Given a source cell and a destination cell, is this command legal now?
Responsibilities:
Reject moves outside the board.
Reject move attempts from empty cells.
Reject moving to a friendly-occupied destination.
Ask the relevant movement rule whether the destination is legal.
Return a clear validation result.
The common route does not implement check, pins, checkmate, castling, en passant, or promotion. The win condition is king capture.
Result shape:
MoveValidation:is_valid: boolreason: string
reason is always present. For a valid move, it is "ok". For an invalid rule-level move, it is a stable machine-readable reason such as "outside_board", "empty_source", "friendly_destination", or "illegal_piece_move".
RuleEngine is read-only with respect to Board. It inspects board state and returns MoveValidation, but it does not move pieces, remove captures, start motions, or update game state.
Game-over is handled by GameEngine, not RuleEngine. GameEngine.request_move checks the game_over flag before calling RuleEngine. If the game is already over, GameEngine returns MoveResult with reason "game_over" and does not call RuleEngine.
The DSL does not assert these reasons. Unit tests assert them.
9. GameEngine Design
GameEngine is the application-service coordinator. It is the public command boundary used by Controller and TextTestRunner.
GameEngine answers these questions:
Is the game already over?
Is another motion already active in the common route?
Should this requested move be delegated to RuleEngine for legality validation?
Should a validated move start a Motion through RealTimeArbiter?
Should simulated time be advanced through RealTimeArbiter?
What read-only GameSnapshot should be exposed to the renderer?
Responsibilities:
Hold or reference the current GameState, including the game_over flag.
Reject request_move when game_over is true, returning MoveResult(reason="game_over").
Reject request_move when the common route already has an active motion, returning MoveResult(reason="motion_in_progress").
Call RuleEngine.validate_move only after application-level guards pass.
Start a legal motion through RealTimeArbiter.
Delegate wait(ms) to RealTimeArbiter.advance_time(ms).
Receive king-capture notification from arrival resolution and set game_over.
Create GameSnapshot for the renderer and BoardPrinter.
GameEngine must not contain piece-specific movement logic, pixel mapping, rendering code, text parsing, or test-runner logic.
MoveResult:
is_accepted: boolreason: string
For a valid accepted command, reason is "ok". For application-level rejection, common-route reasons include "game_over" and "motion_in_progress". Rule-level invalid reasons are copied from MoveValidation.
10. Real-Time Movement Design
Real-time movement is isolated in RealTimeArbiter. It is not mixed into the board, renderer, or controller.
RealTimeArbiter receives only already validated move commands and updates active motions as time advances.
Active motions are stored outside Board. Board represents logical occupancy only. RealTimeArbiter owns the collection of active Motion objects.
In the common route, RealTimeArbiter exposes whether any motion is active. GameEngine uses that fact to enforce the one-active-motion policy before starting a new motion.
Fixed Constants for Tests
The course uses these constants:
CELL_SIZE = 100 pixelsPIECE_SPEED = 100 pixels per second
Timing is deterministic:
Moving one square takes 1000 ms.
Moving two squares takes 2000 ms.
Moving N squares takes N × 1000 ms.
Diagonal movement uses cell-step duration, not Euclidean pixel distance.
A bishop moving one diagonal square takes 1000 ms. A bishop moving three diagonal squares takes 3000 ms.
Logical Board Update Rule
A moving piece remains logically on its source cell until it arrives. The board occupancy changes only on arrival.
This keeps one authoritative logical board state while still allowing the renderer to draw interpolated piece positions from snapshots.
This rule makes print board deterministic:
Before arrival, print board shows the old board.
At or after arrival, print board shows the updated board.
The view can draw the piece between cells during movement. The logical board still changes only on arrival.
Arrival Rule
When a moving piece arrives:
Remove it from the source cell.
Capture the enemy piece at the destination, when one exists.
Place the moving piece at the destination.
If the captured piece is a king, report the king capture to GameEngine.
Arrival resolution is atomic from the perspective of the rest of the system: remove from source, resolve capture, place at destination, update piece state, report king capture, and expose one consistent post-arrival state.
Time in Tests
Tests never call real sleep. The engine exposes:
engine.wait(ms)
The text test command wait 1000 calls this method.
GameEngine.wait(ms) delegates time advancement to RealTimeArbiter. It must not directly manipulate Board motion state.
11. Controller Design
The controller translates user actions into game commands. It does not decide chess legality.
Responsibilities:
Receive click coordinates.
Convert pixels to board cells using BoardMapper.
Maintain selected-piece state.
On first click, select a piece.
On second click, call GameEngine.request_move(source, destination).
Clear selection after every second in-board click, whether the move is legal or illegal.
If no piece is selected, ignore clicks outside the board.
If a piece is selected, an outside-board click cancels the selection and sends no command to GameEngine.
Ignore first clicks on empty cells.
This selection policy is fixed so tests remain simple: outside-board clicks are ignored only when there is no selection; when there is a selection, they cancel it.
The controller must not call Board.move_piece directly. It also must not call RuleEngine directly unless GameEngine intentionally exposes a validation-only query for UI feedback.
Pixel-to-Cell Mapping
With CELL_SIZE = 100:
col = x // 100row = y // 100
Examples:
click 50 50 -> row 0, col 0click 150 50 -> row 0, col 1click 50 150 -> row 1, col 0
A click is outside the board when the mapped row or column is outside board bounds.
Camera / Viewport Decision
The common route has no scrolling camera. Pixel coordinates map directly to board coordinates.
Viewport support belongs only to an extra route. If added, the mapping is implemented inside BoardMapper, not inside the model.
12. View / Renderer Design
The view draws the game state. It does not own game rules and does not mutate the board.
The supplied drawing/image library is wrapped behind a small renderer interface.
Renderer responsibilities:
Draw the board grid.
Draw pieces at pixel positions.
Highlight the selected piece.
Draw moving pieces between cells.
Show a game-over message.
The renderer receives a read-only snapshot:
GameSnapshot:board_widthboard_heightpieces with kind/color/pixel position/stateselected cellgame_over flag
View tests remain minimal. The main test coverage belongs to model, rules, controller, real-time movement, and text integration behavior.
The renderer receives GameSnapshot data only. It does not receive live Board or Piece objects, because live domain objects increase coupling and create opportunities for accidental mutation from the view layer.
13. Text-Based Integration Test Language
The text-based tests are the visible specification of the game. They are also the strongest tool for keeping students focused.
BoardParser and BoardPrinter are text I/O adapters, not test-only helpers. ScriptRunner uses them, but they live outside texttests so the application and tests share the same board setup and board output behavior.
Required Commands
The DSL contains exactly four commands:
Board<textual board rows>click <x> <y>wait <milliseconds>print board
There are no extra DSL assertion commands.
Board Notation
Rules:
Each row is written on a separate line.
Cells are separated by spaces.
. means an empty cell.
White pieces use prefix w.
Black pieces use prefix b.
Piece letters use chess notation: K, Q, R, B, N, P.
Board width and height are inferred from the text.
All rows must have the same number of cells.
Example:
BoardwK . .. wR . .. . bN .. . . bK
click
click x y simulates a user click in pixels.
Example:
click 150 150
With CELL_SIZE = 100, this clicks row 1, column 1.
wait
wait ms advances simulated game time. It never calls real sleep.
Example:
wait 1000
print board
print board is the only integration assertion mechanism.
After every print board, the script contains the exact expected logical board output. The script runner compares the actual board-printer output to those expected rows.
print board prints logical board occupancy, not animation pixel position.
Example:
Board. wR .. . .. . bKclick 150 50click 150 250wait 2000print board. . .. . .. wR bK
The test passes only when the printed board exactly matches the expected rows after print board.
14. Minimal Assertion Strategy
The integration test suite uses exactly one assertion style:
print board<expected board rows>
This single assertion style covers all required common-route behavior:
Board parsing is tested by printing the initial board.
Pixel mapping is tested by clicking a piece and observing the resulting board after movement.
Legal movement is tested by observing the piece at the destination after enough time.
Partial real-time movement is tested by printing before arrival and after arrival.
Illegal movement is tested by printing an unchanged board.
Blocking is tested by printing an unchanged board after an attempted blocked move.
Capture is tested by printing a board where the captured piece disappeared.
King capture and game-over behavior are tested by attempting another legal-looking move after the king capture and printing an unchanged board.
Selection state is tested only by unit tests. The integration suite does not assert selection directly, because selection is not part of print board.
Animation pixel position is tested only in renderer or snapshot unit tests. The integration suite does not assert animation position.
15. How Text Tests Connect to Implementation
The text test runner does not duplicate game logic. It performs the same actions a user performs:
It also does not own board parsing or board printing rules. It delegates those responsibilities to the shared text I/O adapters.
Board text -> BoardParser -> GameEngine initial stateclick -> Controller.click(x, y)wait -> GameEngine.wait(ms)print board -> BoardPrinter.print(game_state)
The text runner never bypasses the controller for clicks and never bypasses the real-time arbiter for time.
Flow:
ScriptRunnerParses commandCalls Controller for clickCalls GameEngine for waitCalls BoardPrinter for print boardCompares expected output
Forbidden shortcut:
ScriptRunner directly calls Board.move_piece.
This is wrong because it bypasses Controller, GameEngine, RuleEngine, and RealTimeArbiter. The test would no longer prove that the real user command path works.
16. Unit Test Layers
The project requires both unit tests and text integration tests.
Unit Tests Answer: “Is This Component Correct in Isolation?”
Examples:
BoardMapper maps pixels to cells correctly.
RookRule stops at blockers.
Board rejects duplicate pieces in one cell.
RealTimeArbiter completes a one-square move after 1000 ms.
Controller selects a piece on first click and issues a move on second click.
RuleEngine returns stable machine-readable invalid-move reasons.
Text Integration Tests Answer: “Does the Feature Work From the Outside?”
Examples:
Given this board, when the user clicks a rook and clicks a destination, after enough time the printed board changes.
Given a blocked rook path, after the user tries the move, the printed board remains unchanged.
Given a king capture, after the user tries another move, the printed board remains unchanged because the game is over.
For each feature, students write several unit tests and one text integration test.
Unit tests do not replace integration tests. Integration tests do not replace unit tests.

17. Iteration Plan for the Teacher
The course is organized as 11 iterations of 7 teaching hours each.
Each iteration ends with passing tests, a small demo, and a reviewed commit.
At the start of each iteration, students first write their own design plan: affected classes, responsibility boundary, tests to add, and expected public behavior.
Only after writing that plan do students ask the LLM to critique it for missing edge cases, responsibility leakage, over-engineering, and missing tests.

AI task: Ask the LLM which state must remain outside Board during motion and which tests prove Board still represents only logical occupancy.
Required behavior:

A two-cell move takes 2000 ms.
Before arrival, printed logical board remains unchanged.
After arrival, printed board updates.
While one motion is active, another requested move is rejected with reason "motion_in_progress".

GameEngine rejects a second move while RealTimeArbiter reports an active motion.

Friendly piece at destination makes the command invalid.

After game over, further clicks do not change the board.

Unit tests:

19. Common Mistakes and Teacher Corrections
Mistake: Putting Pixel Coordinates in the Model Correction: Pixels belong in BoardMapper, renderer, and animation snapshots. The model uses cells.
Mistake: Putting Chess Rules in the Controller Correction: The controller only translates user intent. The RuleEngine decides legality.
Mistake: Making Tests Sleep in Real Time Correction: Tests use wait(ms) to advance simulated time.
Mistake: Implementing All Pieces Before Testing One Piece Well Correction: Start with rook. Use it to prove the architecture.
Mistake: Using the UI to Test Logic Manually Correction: Every visible behavior is reproducible through a text test.
Mistake: Relying on the LLM to Design Everything Correction: Students ask the LLM for alternatives, tests, and edge cases. Students decide and test.
Mistake: Asking the LLM for the full answer when stuck Correction: Ask for a clue, a failing test, the likely layer where the bug belongs, or the next diagnostic question. Students still write and verify the implementation.
Mistake: Making print board Depend on Rendering Correction: print board is generated from logical game state, not screenshots.
Mistake: Adding Many DSL Assertions Correction: The common course uses only print board followed by expected rows. Other state is tested with unit tests.
Mistake: Making GameEngine the main object Correction: GameEngine coordinates Board, RuleEngine, and RealTimeArbiter, but it does not contain piece-specific movement logic, rendering code, input parsing, DSL parsing, or pixel mapping.
Mistake: Letting RuleEngine mutate Board Correction: RuleEngine validates only. GameEngine and RealTimeArbiter apply state changes at the correct time.
Mistake: Passing live Board or Piece objects into Renderer Correction: Renderer receives GameSnapshot only and treats it as read-only.
Mistake: Storing active Motion objects inside Board Correction: Board owns logical occupancy. RealTimeArbiter owns active motion state.
Mistake: Hiding BoardParser inside texttests Correction: BoardParser and BoardPrinter are shared text I/O adapters. The text test runner uses them but does not own them.
Mistake: Starting a second common-route move while another piece is moving Correction: Common route has one active motion at a time. GameEngine rejects the second request with reason "motion_in_progress".
Mistake: Treating an outside-board second click as both ignored and selection-clearing Correction: If no piece is selected, outside-board clicks are ignored. If a piece is selected, an outside-board click cancels selection and sends no command.
Mistake: Testing arrival before real-time movement exists Correction: Iteration 4 tests command routing and validation only. Iteration 5 introduces wait(ms), active motions, and arrival.
20. Minimal Public API Between Layers
The project exposes this API between layers:
BoardParser.parse(text) -> BoardBoardPrinter.print(game_state or snapshot) -> textBoardMapper.pixel_to_cell(x, y) -> Position or NoneController.click(x, y) -> ControllerResultGameEngine.request_move(source, destination) -> MoveResultGameEngine.wait(ms) -> NoneGameEngine.snapshot() -> GameSnapshotRuleEngine.validate_move(board, source, destination) -> MoveValidationRealTimeArbiter.has_active_motion() -> boolRealTimeArbiter.start_motion(piece, source, destination) -> NoneRealTimeArbiter.advance_time(ms) -> ArrivalEvents
This API is intentionally small. Small interfaces are easier to test.
GameEngine.request_move checks game_over and motion_in_progress before delegating to RuleEngine. RuleEngine does not know about either application-level condition.

21. Definition of Done
A feature is done only when:
Unit tests cover the internal logic.
At least one text integration test covers the user-visible behavior through print board.
The implementation does not introduce UI dependency into the model.
Tests do not use real time.
The teacher can read the diff and understand the responsibility of each changed class.
The student can explain what they asked the LLM, what they accepted, and what they rejected.
The student can show the plan they wrote before asking the LLM and the tests used to verify any AI-assisted change.
22. Final Teacher Guidance
The project feels like building a game, but the course is really about professional software construction.
The teacher repeatedly asks:
Where does this responsibility belong?
How do we test this before seeing the UI?
What is the smallest failing test?
What could the LLM have missed?
What breaks when we add another piece type?
What breaks when we add animation, cooldown, or networking?
A good student solution is not the one with the most features. A good solution is the one where a new rule is added with a small test, a small implementation, and a small reviewable diff.
